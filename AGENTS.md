# AGENTS.md

## 1. 目的与适用范围

本文件用于约束代理在 `zt-react-milkdown` 仓库内的协作方式，确保改动聚焦、可解释、低歧义。

适用范围：

- 仓库根目录及所有子目录。

规则优先级：

1. 用户显式需求（最高优先级）
2. 本文件约束
3. 通用工程建议

## 2. 核心强制项（comment-first-tsjs）

当任务涉及 TS/JS 代码文件（`.ts`、`.tsx`、`.js`、`.jsx`、`.mjs`、`.cjs`）改动时，必须使用 `comment-first-tsjs` skill。

执行时至少遵循以下规则：

- 仅补齐缺失注释，不重写已有有效注释。
- 新增注释默认使用中文。
- 函数定义与枚举定义使用 `/** ... */` 风格注释。
- 变量定义与枚举成员使用 `// ...` 风格注释。

## 3. Do / Don’t

### Do

- 先阅读与任务直接相关的文档、源码与配置，再实施改动。
- 坚持最小改动原则，只修改与当前需求直接相关的内容。
- 在交付中清楚说明改动了哪些文件，以及行为变化是否存在风险。

### Don’t

- 不做与当前需求无关的重构、依赖调整或目录结构改造。
- 不擅自扩展需求范围、公共接口或业务行为。
- 不以“规范优化”为由改写已有有效注释或大面积改动无关文件。
- 不执行任何开发命令。

## 4. 交付要求（免强制检验命令）

每次任务完成后，交付说明至少包含：

- Changed：已修改文件路径与变更要点。
- Risk：潜在兼容性或行为变化风险；若无明显风险，明确写“无新增高风险项”。

说明：

- 本仓库不在本文件中设置“必须执行”的固定检验命令。
- 可根据任务需要选择性执行校验，并在交付中说明是否执行及原因。

## 5. 项目目录约定

为保证改动“放对位置、最小改动、避免跨层污染”，新增或调整文件时遵循以下约定。

### 根目录职责

- `src`：生产代码唯一入口。
- `tests`：单元/集成测试，不承载业务实现源码。
- `examples`：示例与联调工程，不回填生产实现。
- `docs`：文档说明，不放可执行业务逻辑。
- `scripts`：脚本与自动化工具，不承载运行时业务模块。
- `dist`、`node_modules`、`.pnpm-store`、`.npm-cache`：构建/依赖产物，禁止手工改动。

### `src` 子目录职责

- `src/core`：编辑器核心能力与基础逻辑。
- `src/plugins`：插件扩展实现。
- `src/react`：React 侧封装与组件接入层。
- `src/styles`、`src/theme`：样式与主题资源。
- `src/types`：公共类型声明。
- `src/utils`：通用工具函数（仅放可复用、与业务无强耦合逻辑）。

### 放置规则

- 新文件优先放入现有职责最匹配目录，不新增目录层级除非确有必要。
- 不将示例代码、测试辅助逻辑放入 `src` 生产目录。
- 不因“顺手优化”移动无关文件；目录调整需有明确需求驱动。

## 6. 样式组织规则（CSS）

- 模块新增样式必须放在模块独立 css 文件中，按职责就近放置（例如 `src/styles/modules/*.css`）。
- `src/styles/style.css` 仅用于全局基础样式与模块样式导入，不直接编写模块细节样式。
- 修改某个模块样式时，优先修改对应模块 css，避免在全局样式文件产生无关改动。
